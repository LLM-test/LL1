# День 12: Персонализация ассистента

## Задание

Добавить систему профиля пользователя: 6 параметров (имя, уровень, стиль/формат ответов, язык, ограничения), которые автоматически инжектируются в system prompt агента при каждом запросе. Профиль должен влиять на тон, детальность и формат ответов — без изменения самого промпта вручную.

---

## Что реализовано

### Domain
- **`UserProfile`** — data class с 6 полями + вычисляемый `isEmpty`
- **`ProfilePresets`** — 3 готовых пресета: `JUNIOR`, `SENIOR`, `STUDENT`
- **`ProfileRepository`** — интерфейс: `getProfile(): Flow`, `setProfile()`, `clearProfile()`, `getProfilePrompt(): String`

### Data
- **`UserProfileEntity`** — Room @Entity, одна запись (id = 1 всегда)
- **`ProfileDao`** — `@Upsert`, Flow-запрос, синхронный запрос, deleteAll
- **`ProfileRepositoryImpl`** — форматирует блок профиля для system prompt, пропускает пустые поля

### Agent
- **`Agent`** — добавлен параметр `profileRepository`; в `buildContextMessages()` профиль инжектируется **до** блока памяти

### Presentation
- **`ProfileScreen`** — 3 пресет-чипа + 6 `OutlinedTextField` + кнопка «Сохранить» + live preview system prompt
- **`ProfileDemoScreen`** — 3 guided-сценария с пошаговой визуализацией и before/after сравнением

## Ключевые технические решения

### 1. Single-record Room + @Upsert
Профиль — всегда одна запись с `id = 1`. Вместо INSERT + UPDATE используется `@Upsert` — одна аннотация покрывает оба случая. `getProfileSync()` нужен только для `getProfilePrompt()`, где нельзя использовать Flow (suspend-контекст внутри Agent).

### 2. Порядок инжекции в system prompt
```
[базовый system prompt]
[=== ПРОФИЛЬ ПОЛЬЗОВАТЕЛЯ ===]   ← идёт первым (персонализация важнее контекста)
[=== ПАМЯТЬ АССИСТЕНТА ===]      ← идёт вторым (рабочий/долговременный контекст)
```
Порядок важен: модель читает инструкции сверху вниз, поэтому профиль задаёт «кто такой пользователь», а память — «что сейчас происходит».

### 3. Пресеты как baseline + copy()
`ProfilePresets.JUNIOR/SENIOR/STUDENT` — объекты-константы. В демо сценарий 3 использует `ProfilePresets.SENIOR.copy(constraints = "...")` — чистый способ изменить одно поле без пересоздания всего объекта.

### 4. Fair comparison в демо
Перед каждым chat-шагом вызывается `agent.reset()` внутри `runProfileStep()`. Это обнуляет историю диалога, чтобы предыдущий ответ (без профиля) не влиял на следующий (с профилем) через контекст окна.

### 5. Live preview в ProfileScreen
`getProfilePrompt()` вызывается после каждого `save()` / `clear()` и обновляет `systemPromptPreview` в State. Пользователь сразу видит, как изменится блок в system prompt, — без необходимости переходить в AgentScreen.

## Блок профиля в system prompt

Пример для пресета JUNIOR:
```
=== ПРОФИЛЬ ПОЛЬЗОВАТЕЛЯ ===
Уровень: Junior Developer
Стиль ответов: подробный и понятный
Формат ответов: с пошаговыми объяснениями и примерами
Язык: Русский
Ограничения: объясняй каждый шаг, не пропускай детали
```
Пустые поля не добавляются. Если профиль полностью пустой — блок не инжектируется вовсе.

## Вывод

Профиль — это «мета-контекст» пользователя, который живёт постоянно (в Room) и прозрачно накладывается на каждый запрос. В отличие от памяти (которая формируется динамически из диалога), профиль задаётся один раз и описывает **кто** задаёт вопросы, а не **о чём** они. Совместно с памятью (День 11) система теперь знает и контекст задачи, и контекст пользователя — два разных измерения персонализации, работающих независимо и не мешающих друг другу.
